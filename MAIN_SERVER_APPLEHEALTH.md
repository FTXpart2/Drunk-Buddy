# Main Server Setup — Apple Health Heart-Rate Guardian

Instructions for the teammate running the main agent server. Goal: get real Apple
Watch heart rate flowing into the buddy so it proactively checks in (and escalates)
during party mode. Everything below is run from the repo root in your terminal.

The feature is on branch **`Debu`**. If you're on `main`, none of the vitals code
(`/vitals`, `/watch`, the guardian, `get_health_link`) exists yet — step 1 is required.

---

## 1. Get the code

```bash
git fetch origin
git checkout Debu          # or merge Debu into your branch
git pull origin Debu
```

## 2. Install dependencies

This is a pnpm monorepo. Do **not** run `npm install` (it corrupts the pnpm
`node_modules`). If you don't have `pnpm`, use `npx`:

```bash
npx -y pnpm@latest install --config.minimumReleaseAge=0
```

(The `--config.minimumReleaseAge=0` flag bypasses a supply-chain age check that
otherwise blocks a freshly published transitive dep.)

## 3. Configure `.env`

```bash
cp .env.example .env        # only if you don't already have a .env
```

Then set these keys in `.env`:

```
ANTHROPIC_API_KEY=sk-ant-...            # your real Anthropic key (starts sk-ant-)
ANTHROPIC_MODEL=claude-haiku-4-5

# heart-rate guardian
HEALTH_LINK_SECRET=<any-random-string>  # signs the health link token; pick something
HR_HIGH=80                              # DEMO: low so a resting reading triggers (prod ~130)
HR_LOW=45
ESCALATE_AFTER_MS=15000                 # DEMO: escalate 15s after an unanswered check-in
HEARTBEAT_MS=30000

# filled in at step 5
PUBLIC_URL=
```

Notes:
- `HEALTH_LINK_SECRET`: any value, but once set, **only links generated by this
  server will verify**. That's fine — the buddy generates them (step 6). Don't paste
  tokens from another machine.
- `HR_HIGH=80` and `ESCALATE_AFTER_MS=15000` are demo-friendly. Raise for real use.

## 4. Start the server

```bash
node_modules/.bin/tsx backend/src/index.ts
```

Leave this terminal running. It serves the agent loop **and** the `/vitals` +
`/watch` endpoints on port `8787`. For live iMessage instead of the terminal
channel, set `CHANNEL=bluebubbles` (plus the BlueBubbles vars) — see the main README.

## 5. Expose the server to the internet (new terminal tab)

The phone can't reach `localhost`. Open a **second terminal** and start a tunnel.
No-account option:

```bash
brew install cloudflared
cloudflared tunnel --url http://localhost:8787
```

Or ngrok (needs a free account + `ngrok config add-authtoken <token>`):

```bash
brew install ngrok
ngrok http 8787
```

Copy the `https://…` URL it prints. Put it in `.env` and **restart the server**
(Ctrl+C in the step-4 terminal, then rerun):

```bash
# in a spare terminal:
sed -i '' 's|^PUBLIC_URL=.*|PUBLIC_URL=https://YOUR-TUNNEL-URL|' .env
```

> The free tunnel URL changes every time you restart it. If it changes, update
> `PUBLIC_URL`, restart the server, and have users grab a fresh link (step 6).

Sanity check the tunnel reaches the server (new terminal):

```bash
curl -s https://YOUR-TUNNEL-URL/health
# -> {"ok":true,"channel":"local","model":"claude-haiku-4-5"}
```

## 6. Each user gets their own link from the buddy

Do **not** hand-paste tokens. The token encodes the user's phone (the join key) and
is signed with *this* server's secret. The buddy hands it out automatically:

1. The user texts the buddy at least once (this saves their reply target so the buddy
   can message them unprompted).
2. The user tells the buddy they're heading out → the buddy turns on **party mode**
   and texts the one-tap link:
   `https://YOUR-TUNNEL-URL/watch?t=<token>`

For the iOS Shortcut (step 7), the user POSTs to the **same URL with `/watch`
swapped for `/vitals`**, keeping the full `t=` token (everything including the part
after the `.`).

## 7. On the user's iPhone — the Shortcut (one-time, per user)

Shortcuts app → **+** → add these actions:

1. **Find Health Samples** — Type `Heart Rate`, Sort `End Date`, Order `Latest
   First`, Limit `1`.
2. **Get Item from List** — `First Item`.
3. **Get Contents of URL**
   - URL: `https://YOUR-TUNNEL-URL/vitals?t=<FULL_TOKEN>` (plain text, no blue
     variable chips — a chip causes a "couldn't convert Rich Text to URL" error)
   - Method: **POST**
   - Header: `Content-Type` = `application/json`
   - Request Body: **JSON**, add a **Number** field, Key = `hr`, Value = the sample's
     **Value** property (tap the inserted variable → choose `Value`, not the whole
     sample — otherwise the server sees `hr:0`)

Run it once (▶), allow Health + network access. For a real ~15-min background cadence,
use the **Health Auto Export** app (App Store) pointed at the same `/vitals?t=<token>`
URL — iOS automations can't reliably fire every 15 min on their own.

## 8. Verify end to end

Watch the server terminal while the Shortcut runs:
- `vitals.reject {"hasPhone":false}` → the token is truncated/wrong (copy the full
  `t=`, including after the `.`).
- `vitals.reject {... "hr":0}` → the `hr` field isn't the numeric **Value** (fix the
  Shortcut body).
- No reject line → accepted. With **party mode on** and HR over `HR_HIGH`, the buddy
  texts "you ok?"; stay silent past `ESCALATE_AFTER_MS` and it fires `alert_circle`.

The Apple Watch must be on-wrist and have logged a recent HR (it syncs to the Health
app), or "Find Health Samples" returns nothing.

## Gotchas / things that silently make it do nothing

- **Wrong branch** — no vitals code on `main`. Must be on `Debu`.
- **Party mode off** — the guardian only watches during party mode. User must tell the
  buddy they're heading out.
- **User never texted the buddy** — no saved reply target, so the buddy can't reach
  them. They must message once first.
- **Tunnel URL changed** — old links point at a dead host. Update `PUBLIC_URL`,
  restart, re-issue links.
- **In-memory state** — party mode / vitals / reply target reset on server restart.
  Set `REDIS_URL` to persist across restarts for a multi-user demo.
- **Token from another machine** — only verifies against the server whose
  `HEALTH_LINK_SECRET` signed it. Always let the buddy generate the link.

## Separate setup you also own: The Token Company (prompt compression)

Unrelated to Apple Health, but it's your server, so it's your key to add. The agent
can route Claude calls through The Token Company, which ML-compresses the system
prompt + user messages before each call (cheaper/faster). It's **off unless you set a
key**, and **I could not fully verify it** because it needs a real `ttc-...` key I
don't have. Action for you:

1. Get a compression key (`ttc-...`) from the Token Company dashboard.
2. Add it to `.env` (the Anthropic key is still used for the actual call):
   ```
   ANTHROPIC_API_KEY=sk-ant-...
   TOKEN_COMPANY_API_KEY=ttc-...
   ```
3. Restart the server. On boot the log should read `llm.compress {"via":"token-company"}`
   instead of `llm.anthropic`. If it still says `llm.anthropic`, the key isn't being
   picked up.
4. Send one message through the buddy and confirm a normal reply. If it errors, the key
   is the likely cause — the wiring matches their official Node sample
   (`withCompression(new Anthropic(), { compressionApiKey })` from
   `the-token-company/anthropic`), so a valid key should just work.

Notes:
- The dependency `the-token-company` is already installed (in `backend`); no install
  step needed.
- Leave `TOKEN_COMPANY_API_KEY` unset to disable compression entirely — the agent falls
  back to calling Anthropic directly. Nothing else changes.
- It's a pure add-on to the LLM call; it does not touch the Apple Health / guardian flow
  above.ca
